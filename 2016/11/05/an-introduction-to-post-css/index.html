PostCSS 안내서

원문 바로가기: An Introduction To PostCSS by Drew Minns

다른 개발 언어와 마찬가지로 CSS 역시 순환적인 발전 과정을 거칩니다. 매번 주요 배포 때마다 새로운 기능과 구문이 등장합니다. CSS 레벨 3에서 새로 배포된 기능 중 인터렉션과 관련된 것들은 예전에는 자바스크립트만을 사용하여 구현할 수 있었습니다. 하루가 멀다하고 새로운 툴이 등장하고, 그 덕분에 웹 페이지 스타일링 작업은 점점 더 쉬워지고 있습니다.

PostCSS는 비교적 새로 등장한 스타일링 툴 중 하나입니다. PostCSS의 목표는 커스텀 플러그인과 툴로 이루어진 개발 환경을 도입하여 CSS 개발과정에 혁신을 일으키는 것입니다. PostCSS의 원칙은 Sass, LESS 같은 전처리기의 원칙과 마찬가지로 확장된 구문과 기능을 브라우저에 친화적이고 현대적인 CSS로 변환하는 것입니다.

뭐로 하는지 궁금하다구요? 자바스크립트를 사용하는 것입니다.

자바스크립트는 다른 어떤 처리기보다 스타일 변환을 빠르게 할 수 있습니다. Gulp와 Grunt같은 태스크러너 도구를 사용하면 스타일 변환은 빌드 과정에서 이루어지는데, 이는 Sass와 Less 컴파일 과정과 비슷합니다. React나 AngularJS와 같은 라이브러리 및 프레임워크를 사용한다면 자바스크립트 코드에 CSS를 직접 써 넣을 수 있으므로 자바스크립트를 통해 스타일 변환을 할 수 있습니다.

PostCSS의 역사

Autoprefixer를 개발한 안드레이(Andrey Sitnik)는 자바스크립트를 CSS 처리에 사용하고자 PostCSS를 만들었습니다. PostCSS 그 자체는 하나의 API에 지나지 않으며, 이미 개발되어 있는 수많은 플러그인과 조합한다면 여러가지 강력한 기능을 사용할 수 있습니다. 디버깅을 손쉽게 할 수 있도록 도와주거나 소스맵을 생성해주며, 내장된 추상 구문 트리(abstract syntax tree, AST)로 코드가 어디서 어떤식으로 변환되는지 알아 볼 수 있습니다.

PostCSS는 Node.js 프레임워크를 사용하기 때문에 손쉽게 내부 코드를 수정하여 기존 툴을 다르게 사용할 수도 있습니다. 이와는 대조적으로 Sass와 LESS같은 툴은 사용자들이 컴파일러 작성 시점의 메소드만을 사용하도록 시스템에 제한을 두었습니다.

PostCSS의 API 덕분에 어떤 기능이든 사용자가 필요로 한다면 플러그인이나 툴을 만들 수 있습니다. 모듈을 바탕으로 디자인된, 중립적인 성격을 띈 툴이기 때문에 프로젝트 요구사항에 따라 필요한 기능에 초점을 맞추어 작업할 수 있습니다. PostCSS는 어떤 특정 언어 형식에 종속되어 있지 않으며, 필요하다면 Sass와 LESS같은 전처리기의 구문 스타일을 사용할 수 있습니다.

모듈 단위로 생각했을 때의 장점

대부분의 개발자들이 프로젝트를 맨땅에서 시작하는 경우는 드뭅니다. 많은 사람들이 변수, 믹스인, 함수 등을 담고 있는 Sass 보일러플레이트로 프로젝트를 시작합니다. 스타일 시트를 함수, 믹스인, 그리드 시스템에 따라 여러 파일로 분리하고 개발을 좀 더 손쉽게 진행할 수 있는 유용한 코드를 담은 파일도 따로 마련해 놓습니다. 이런 작업을 다 하고 나면 스타일 시트 파일은 보통 10개 정도 생성됩니다.

Sass와 LESS 코드를 모아놓은 라이브러리는 관리하다보면 어느샌가 무리를 느끼고 있고, 프로젝트의 규모 또한 불어나게 됩니다. 많은 프로젝트에서 “만약을 대비한” 코드가 포함된 사용하지 않는 믹스인이나 함수를 흔히 발견할 수 있습니다. PostCSS를 사용한다면 플러그인을 모듈 단위로 손쉽게 설치할 수 있기 때문에, 프로젝트 요구사항에 맞춰서 개발 과정을 변경할 수 있습니다.

PostCSS를 사용하면 개발 중인 스타일 시트에서 함수, 유틸리티, 믹스인을 구성하는 코드만을 따로 빼내어 플러그인으로 만들 수 있습니다. 개별 프로젝트마다 태스크 러너에 플러그인을 포함시켜서 필요한 툴만 사용할 수 있습니다.

PostCSS FontPath 플러그인 예시로 좀 더 자세하게 알아봅시다. 만약 Sass를 사용한다면 파일 안에 웹 폰트에 대한 사용자 설정을 할 수 있는 믹스인을 만들어야 하겠죠. 이 믹스인 내에는 @font-face태그가 들어갑니다.

PostCSS FontPath를 사용한다면 위의 코드처럼 믹스인을 써야 할 필요가 없습니다. 그저 아래와 같은 CSS만 작성해 놓으면 PostCSS가 이를 Grunt나 Gulp를 통해 최종 산출물로 변환할 것입니다.

이 포스팅을 쓰고 있는 현재 시점에서 본다면 PostCSS 관련 플러그인이 100개가 넘으며 이들이 제공하는 기능을 사용하여 미래에 나올 CSS 구문이나 단축키, 툴, 언어 확장판 등을 만들어 볼 수 있습니다. PostCSS는 그저 “쿨한 툴”이 아니라 이미 워드프레스, 구글, 트위터 팀에서 개발의 토대로 사용하고 있는 툴입니다.

플러그인 설치하기

PostCSS 사용의 완성은 플러그인에 달려 있습니다. 지금까지 본 Gulp와 Grunt 코드 안에 있는 태스크 선언문에 빈 배열이 있는 것을 눈치 채셨을 것입니다. 이 배열 안에 PostCSS 커뮤니티에서 개발한 플러그인을 넣어서 원하는 기능을 툴에 포함할 수 있습니다.

PostCSS 깃허브 페이지를 보면 사용 가능한 PostCSS 플러그인 목록을 보실 수 있습니다. 모든 NPM 패키지와 마찬가지로 플러그인 또한 커맨드 라인에서 설치가 가능합니다. 많은 수의 플러그인이 PostCSS 내에서 확장자로만 사용 가능하며, 태스크 러너 종류에 구애받지 않습니다. 예시로 PostCSS Focus 플러그인을 설치하여 모든 hover 상태에 :foucs를 추가해 봅시다.

다음 예시에 등장하는 플러그인은 전부 커맨드 라인과 NPM을 사용하여 프로젝트에 해당 패키지를 설치해야 사용할 수 있습니다.

PostCSS 플러그인 설치 예제

npm i postcss-focus -D

플러그인

플러그인 설치 및 코드 컴파일을 위한 빌드 툴 준비를 마쳤다면, PostCSS와 플러그인 기능을 사용할 수 있습니다. 가장 먼저 해야할 일은 .css 확장자 파일을 개발 코드 폴더에 생성하는 것입니다.

“잠깐! CSS 파일이라니요?” 넵, CSS 파일 맞습니다. PostCSS가 변형하는 것은 CSS이기 때문에 다른 특별한 구문을 사용할 필요가 없습니다. 그냥 고전적인 CSS면 됩니다. 전처리기에 익숙하다면 .sass, .scss, .styl, .less 확장자를 사용하지 않은 채 .css를 곧장 사용하는 것이 부자연스럽게 느껴질 수도 있습니다. 근데 실제로는 이러한 전처리기 사용 역시, 파일 자체를 다른 파일로 변환하는 것(convert)이 아니라, 안의 내용물인 코드만 바꾸는 것(transform) 뿐입니다.

새로 설치한 PostCSS 플러그인으로 스타일 변환을 진행하려면 grunt styles나 gulp styles를 입력하여 태스크 러너를 돌리면 됩니다. PostCSS 플러그인은 개발 CSS 파일을 변환시켜서 코드에 명시해 놓은 폴더로 배포 CSS를 산출해 냅니다.

아래 플러그인 목록은 이제 막 PostCSS를 사용하기 시작했다면 도움이 될 만한 것들을 모아놓은 것입니다. 플러그인 사용법과 설치 방법을 적어 놓았습니다.
